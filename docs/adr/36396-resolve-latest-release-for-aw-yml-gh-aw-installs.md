# ADR-36396: Resolve Latest Release for Unversioned aw.yml gh-aw Package Installs

**Date**: 2026-06-02
**Status**: Draft

## Context

The `add` and `add-wizard` commands install workflows either as single-workflow specs or as `aw.yml` repository packages. When no `@ref` was supplied, the two paths diverged: single-workflow installs already resolved to the latest stable release, but `aw.yml` package installs resolved straight to the repository default branch. For the canonical `github/gh-aw` package this meant unversioned installs pulled unreleased `main` content instead of the latest stable tag, and the ref used to fetch the manifest was not guaranteed to match the ref later used for downstream spec generation and SHA pinning. The fix must keep behavior unchanged for arbitrary third-party packages while bringing the `github/gh-aw` package into parity with single-workflow resolution.

## Decision

We will resolve the latest stable release tag first for unversioned `github/gh-aw` package installs, falling back to default-branch resolution only when the release lookup is unavailable. The resolved ref is recorded on the package metadata (`resolvedRepositoryPackage.ResolvedRef`) and propagated through `appendRepositoryPackageWorkflowSpecs` via a single `repositoryPackageEffectiveRef` helper, so manifest fetch, workflow/skill/agent spec expansion, and SHA pinning all operate on the same ref. An explicit user-provided `repoSpec.Version` always takes precedence. The `github/gh-aw` special case is identified by an exact, case-insensitive, whitespace-tolerant matcher (`isGhAwRepository`) against a centralized slug constant rather than scattered string literals.

## Alternatives Considered

### Alternative 1: Apply latest-release-first resolution to all repository packages
Treat every package source like single-workflow installs and attempt a release lookup regardless of repository. Rejected because arbitrary packages may not publish releases at all, the extra API call would penalize every install, and changing the default-branch contract for third-party packages is a broader behavioral change than this PR intends. Scoping to `github/gh-aw` limits blast radius.

### Alternative 2: Keep default-branch resolution for package installs
Leave `aw.yml` installs pinned to the default branch and accept the divergence from single-workflow behavior. Rejected because it leaves unversioned `github/gh-aw` installs pulling unreleased content and keeps fetch/pinning refs potentially inconsistent, which is the defect this change exists to fix.

## Consequences

### Positive
- Unversioned `github/gh-aw` `aw.yml` installs now match single-workflow latest-release behavior, giving stable, release-pinned installs.
- A single effective-ref helper guarantees manifest fetch, spec expansion, and SHA pinning all use the same resolved ref.

### Negative
- Introduces a `github/gh-aw`-specific code path (slug constant + matcher), adding maintenance surface and asymmetry versus other packages.
- Adds an extra releases-API call on the unversioned `github/gh-aw` install path.

### Neutral
- Behavior for third-party packages and for explicitly versioned installs is unchanged.
- When the release lookup fails, resolution transparently falls back to the previous default-branch behavior.
